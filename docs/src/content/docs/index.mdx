<h2>The problem</h2>
<p>
Cloud bills double overnight. Privacy audits land badly. Engineers ship slower because every
workload reinvents the same governance from scratch. By the time anyone notices, the workloads,
the habits, and the architecture are already set, and remediation is painful.
</p>

<p>
This pattern repeats with every transformative technology. Privacy gets addressed first because
regulation forces the conversation. FinOps gets addressed last, after a billing shock. AI is
next, and this time the spend is moving faster than the regulation. The numbers tell the story.
</p>

  <div class="stat-card">

<p>
The Data Landing Zone is engineered to prevent this failure mode before the first workload lands.
</p>

<h2>Built for three stakeholders at once</h2>
<p>
Most data platforms optimise for one stakeholder, tolerate the second, and ignore the third. The
DLZ is designed to make all three genuinely happy at the same time.
</p>

<div class="actor-grid">
  <div class="actor-card">
    <div class="actor-icon">💰</div>

  </div>

    </p>
  </div>
  <div class="actor-card">

    </p>

<blockquote class="design-principle">
<strong>Design principle.</strong> Governance should be the path of least resistance. An engineer
using the DLZ should find it easier to do the right thing than to bypass it. Not because we block
them, but because we made the right thing the default.
</blockquote>

<h2>How the DLZ delivers on FinOps</h2>
<p>
FinOps is the pillar that most consistently gets left behind. In the DLZ it doesn't. The
capability is delivered in four layers, each building on the one below.
</p>

<div class="layers-grid">
  <div class="layer-card">
    <div class="layer-number">01</div>
    <h3>Instrumentation</h3>
    <p>
    Every service emits cost and usage data in a consistent, queryable form. CUR 2.0 is the master
    ledger, centralised in a dedicated FinOps account and queryable from Athena.
    </p>
  </div>
  <div class="layer-card">
    <div class="layer-number">02</div>
    <h3>Attribution</h3>
    <p>
    A mandatory tag set is enforced at deploy time through tag policies, SCPs, and DLZ validation.
    Resources that cannot be attributed to an owner do not deploy to production.
    </p>
  </div>
  <div class="layer-card">
    <div class="layer-number">03</div>
    <h3>Guardrails</h3>
    <p>
    Soft alerts at threshold, hard blocks at limit. An LLM gateway enforces per-team and
    per-project token caps in real time, blocking requests before the model ever sees them.
    </p>
  </div>
  <div class="layer-card">
    <div class="layer-number">04</div>
    <h3>Reporting</h3>
    <p>
    Dashboards for engineers, product owners, and finance. Cost-per-unit-of-work makes
    optimisation conversations concrete: cost per inference, per document, per query.
    </p>
  </div>
</div>

## What is the Data Landing Zone?

<CardGrid stagger>
	<Card title="Accelerator" icon="rocket">
    A landing zone tailored for AI, data-driven, data mesh, and cloud projects, so teams can experiment fast without building governance from scratch.
	</Card>
  <Card title="Knowledge baked in" icon="add-document">
    Operational experience, regulatory awareness, and cost discipline are encoded directly in the platform. They travel with every workload built on top of it.

		A CDK construct published in TypeScript and Python that orchestrates AWS accounts and the resources needed for a multi-account data platform.
	</Card>
	<Card title="FinOps as a first-class pillar" icon="seti:db">
    Mandatory cost-allocation tags enforced at deploy time. Budgets and anomaly controls as code. CUR centralised in a dedicated FinOps account. An LLM gateway that blocks token spend before it reaches the model.
	</Card>
	<Card title="Standards &amp; best practices" icon="setting">
		Aligned with the AWS Well-Architected Framework for multi-account strategy, security, and compliance.
	</Card>
  <Card title="Ease of use" icon="heart">
