<h2>A pattern we keep seeing</h2>
<p>
Every time a transformative technology arrives — cloud in 2010, big data in 2014, AI in 2023 —
the same two governance gaps appear in the same order. <strong>Privacy</strong> gets addressed
first, because regulation provides the forcing function. <strong>FinOps</strong> gets addressed
last, and usually too late — only after a billing shock forces the conversation. By then, the
workloads, the habits, and the architecture are already set. Remediation is painful.
</p>

<p>
98% of FinOps practitioners now manage AI spend — up from 31% in 2024. Average monthly AI
spend is projected at ~$85K in 2025, a 36% year-over-year increase. Yet only 51% of teams
feel confident measuring AI ROI.
</p>

<p>
This is the precise failure mode the Data Landing Zone is engineered to prevent.
</p>

<h2>Three actors, one platform</h2>
<p>
Most data platforms optimise for one stakeholder, tolerate the second, and ignore the third.
The DLZ is designed to make all three genuinely happy at the same time.
</p>

<div class="container" style="margin-bottom: 2rem;">
  <div class="column">
    <p><strong>💰 CFO &amp; Finance:</strong> Cost controls are not bolted on after the fact. Every workload is financially observable from day one. Budgets and guardrails are provisioned as code. No end-of-month surprises, no manual reconciliation, clear chargeback lines.</p>

    <p><strong>📋 Compliance Office:</strong> Built with a compliance mindset from the ground up. GDPR alignment, data classification, access control, and audit trails are DLZ defaults — not workload-level responsibilities. Regulators find what they expect to find.</p>
  </div>
  <div class="column">
    <p><strong>👩‍💻 Engineers:</strong> Built by engineers, for engineers. Developer experience is a first-class priority. Engineers don't cut tickets begging a CCoE to unblock them. Productivity is not sacrificed on the altar of governance.</p>

    <p><strong>🧭 Design principle:</strong> Governance should be the path of least resistance, not an obstacle course. An engineer using the DLZ should find it easier to do the right thing than to bypass it — not because we block them, but because we made the right thing the default.</p>
  </div>
</div>


## What is the Data Landing Zone?

<CardGrid stagger>
	<Card title="Accelerator" icon="rocket">
    A landing zone tailored for AI, data-driven, data mesh, and cloud projects — so teams can experiment fast without building governance from scratch.
	</Card>
  <Card title="Knowledge baked in" icon="add-document">
    Operational experience, regulatory awareness, and cost discipline are encoded directly in the platform. They travel with every workload built on top of it.
  </Card>
  <Card title="All-in-one CDK component" icon="pencil">
		A CDK construct published in TypeScript and Python that orchestrates AWS accounts and the resources needed for a multi-account data platform.
	</Card>
	<Card title="FinOps as a first-class pillar" icon="seti:db">
    Mandatory cost-allocation tags enforced at deploy time. Budgets and anomaly controls as code. CUR centralised in a dedicated FinOps account. An LLM gateway that blocks token spend before it reaches the model. Runaway spend is caught before the invoice arrives.
	</Card>
	<Card title="Standards &amp; best practices" icon="setting">
		Aligned with the AWS Well-Architected Framework for multi-account strategy, security, and compliance.
	</Card>
  <Card title="Ease of use" icon="heart">
    Simple enough for Data Engineers to use independently, yet flexible and customisable for Cloud Engineers to fine-tune and optimise.
  </Card>
