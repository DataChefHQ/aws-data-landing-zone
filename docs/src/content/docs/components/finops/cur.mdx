Delivers CUR 2.0 cost data to a dedicated FinOps account. Export lives in the
management account; bucket and Glue catalog live in the FinOps account. AWS Billing
writes Parquet files cross-account.

## Prerequisites

A FinOps account configured under `org.ous.sharedServices.accounts.finOps`. Everything
else is handled by `dlz bootstrap`.

## Minimal example

<DualCode>
  <Fragment slot="ts">

    ```ts
      organization: {
        ous: {
          sharedServices: {
            ouId: 'ou-xxxx',
            accounts: { finOps: { accountId: '123...' } },
          },
        },
      },
      finOps: { cur: {} },

    ```
  </Fragment>
  <Fragment slot="python">

    ```python
            ous=dlz.OrgOus(
    ```

You get:

- BCM Data Exports CUR 2.0 export in the management account (us-east-1).
- S3 bucket `dlz-cur-{finOpsAccountId}-{destinationRegion}` in the FinOps account.
- Glue database `dlz_cur_2` and a daily crawler at 06:00 UTC.
- Athena workgroup `dlz-cur` with a dedicated query-results bucket so queries
  run from day-1 without the "set up a query result location" prompt.
- The 5 mandatory tags activated as Cost Allocation Tags.

## Feature flags

Each `include*Data` flag toggles an `INCLUDE_*` table property. The construct
auto-extends the SELECT column list â€” no manual SQL editing.

| Flag | Default | Adds | Available |
|---|---|---|---|
| `includeResources` | `true` | `line_item_resource_id` | always |
| `includeSplitCostAllocationData` | `true` | `split_line_item_*` | always |
| `includeCapacityReservationData` | `false` | 3 `capacity_reservation_*` columns | from 2025-11-01 |
| `includeIamPrincipalData` | `false` | `line_item_iam_principal` (Bedrock) | from 2026-04-08 |

## Customizing columns

Set `exportConfig.queryColumns` to your own list (falls back to
`DLZ_CUR_DEFAULT_QUERY_COLUMNS` when unset). Flag-driven columns are appended automatically.

To extend the defaults, spread them:

<DualCode>
  <Fragment slot="ts">
    ```ts
    import { DLZ_CUR_DEFAULT_QUERY_COLUMNS } from 'aws-data-landing-zone';

    finOps: {
      cur: {
        exportConfig: {
          queryColumns: [...DLZ_CUR_DEFAULT_QUERY_COLUMNS, 'some_new_column'],
        },
      },
    }
    ```
  </Fragment>
  <Fragment slot="python">
    ```python
    from aws_data_landing_zone import DLZ_CUR_DEFAULT_QUERY_COLUMNS

    fin_ops=dlz.DlzFinOpsProps(
        cur=dlz.DlzCurProps(
            export_config=dlz.DlzCurExportConfig(
                query_columns=[*DLZ_CUR_DEFAULT_QUERY_COLUMNS, "some_new_column"],
            ),
        ),
    )
    ```
  </Fragment>
</DualCode>

For a non-trivial projection / filter / aggregation, set `exportConfig.queryStatement`
to the full SQL. That fully bypasses column composition (you own the SQL).

### S3 path layout

BCM Data Exports always lays files out as
`<bucket>/<destinationPrefix>/<exportName>/{data,metadata}/BILLING_PERIOD=YYYY-MM/...`.
DLZ defaults `destinationPrefix` to empty so the layout collapses to
`<bucket>/<exportName>/{data,metadata}/...` â€” clean, no doubled folders, no
mental overhead.

| `destinationPrefix` | `exportName` | Resulting S3 path |
|---|---|---|
| `''` (default) | `dlz-cur-2` (default) | `<bucket>/dlz-cur-2/{data,metadata}/...` |
| `''` (default) | `acme-cur` | `<bucket>/acme-cur/{data,metadata}/...` |
| `'reports'` | `dlz-cur-2` | `<bucket>/reports/dlz-cur-2/{data,metadata}/...` |
| `'team-a'` | `team-a-cur` | `<bucket>/team-a/team-a-cur/{data,metadata}/...` |

Set `destinationPrefix` only when you need to host multiple exports in one
bucket under separate folders.

### Multi-region

The export resource is always pinned to us-east-1; only the bucket moves via
`destinationRegion`. The Glue crawler auto-pins to the bucket's region.

### Sandbox lifecycle

`dataPlaneConfig.lifecycle.enabled: false` keeps every object in S3 Standard forever.
The construct emits a synth-time warning â€” don't ship this to prod.

### SSE-KMS

Default is SSE-S3. To use a CMK, set `dataPlaneConfig.encryption.kmsKeyArn` and grant
`bcm-data-exports.amazonaws.com` `kms:GenerateDataKey*` and `kms:Decrypt` on the key.
DLZ does not mutate KMS key policies.

## Athena workgroup

DLZ ships an Athena workgroup (`dlz-cur` by default) and a dedicated
query-results bucket so Athena works in the FinOps account without manual
setup. Without it, the AWS console blocks every first query with
`Before you run your first query, you need to set up a query result location in Amazon S3`.

What you get:

- Workgroup with `EnforceWorkGroupConfiguration: true` â€” users cannot override
  the result location or encryption client-side.
- Dedicated results bucket `dlz-cur-athena-results-{accountId}-{region}` with
  BPA, SSL-only access, and a 30-day expiration on results (results are scratch
  data â€” keeping them indefinitely is just cost).
- Encryption mirrors the CUR data bucket (SSE-S3 by default; SSE-KMS with the
  same key when `dataPlaneConfig.encryption.kmsKeyArn` is set).
- Athena engine v3.

### Opting out

If you already run a central Athena workgroup elsewhere, set
`dataPlaneConfig.athena.enabled: false`. No workgroup or results bucket is
created; you wire your own consumers at the Glue catalog.

## Updating the export

The BCM export is owned by a custom resource that calls `UpdateExport` for in-place
changes and falls back to `DeleteExport` + `CreateExport` for changes BCM can't apply
in place. The export name is stable across updates, so the S3 path
(`s3://<bucket>/<exportName>/...`) doesn't move and Glue/Athena consumers see a stable
location.

There's a few seconds of delivery downtime during a fall-back replace; existing data
is unaffected.

## Exported SSM parameters

The CUR data plane writes its state to SSM Parameter Store under
`/dlz/finops/cur/` in the FinOps account. Downstream tooling (CUDOS, Vantage,
Looker, custom dashboards) reads these instead of hard-coding names.

| Parameter | Example value |
|---|---|
| `/dlz/finops/cur/glue-database-name` | `dlz_cur_2` |
| `/dlz/finops/cur/cur-bucket-name` | `dlz-cur-{accountId}-{region}` |
| `/dlz/finops/cur/cur-destination-prefix` | `(none)` (or your `finOps.cur.destinationPrefix` if set) |
| `/dlz/finops/cur/cur-export-name` | `dlz-cur-2` |
| `/dlz/finops/cur/cur-data-path` | `dlz-cur-2` (or `<prefix>/dlz-cur-2` when prefix is set) |
| `/dlz/finops/cur/destination-region` | `us-east-1` |
| `/dlz/finops/cur/athena-workgroup-name` | `dlz-cur` |
| `/dlz/finops/cur/athena-results-bucket-name` | `dlz-cur-athena-results-{accountId}-{region}` |

The Athena entries are only written when `dataPlaneConfig.athena.enabled !== false`.

Reading from a shell:

```bash
aws ssm get-parameter --name /dlz/finops/cur/glue-database-name \
  --query Parameter.Value --output text
# â†’ dlz_cur_2
```

For an external CUDOS / CID deploy, see the
[CUDOS extras page](/components/finops/cudos) for ready-to-paste snippets that
pull these into `cid-cmd` and the AWS CFN template.

## Locked down

- Service principal: always `bcm-data-exports.amazonaws.com`.
- Bucket policy `aws:SourceAccount` and `aws:SourceArn` (`ArnLike` wildcard scoped to
  the management account).
- `BlockPublicAccess: BLOCK_ALL`.
- BCM export region: `us-east-1`.
