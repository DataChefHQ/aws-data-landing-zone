<Aside type="caution" title="Opt-in extra — not deployed by DLZ">
DLZ does **not** ship CUDOS / CID dashboards. Visualization is a separate layer
with a different lifecycle, different pricing model (per-user QuickSight billing),
and a different update cadence than DLZ. This page documents how to deploy CUDOS
**on top of** what DLZ already provisions, and what it costs.
</Aside>

## What CUDOS / CID is

[Cloud Intelligence Dashboards](https://catalog.workshops.aws/awscid/) (CID) is a
collection of QuickSight dashboards — CUDOS, CID Core, KPI, Compute Optimizer,
Trusted Advisor Organizational view — published by AWS in the
[`aws-solutions-library-samples/cloud-intelligence-dashboards-framework`](https://github.com/aws-solutions-library-samples/cloud-intelligence-dashboards-framework)
repository. They sit on top of CUR 2.0 and surface cost, usage, optimization,
and reservation posture without writing SQL.

It is **AWS Solutions Library sample/reference content**, not an AWS GA
service. Updates ship roughly monthly and are best deployed via the `cid-cmd`
Python CLI (the canonical path) or the bundled CFN template.

## Why CUDOS lives outside DLZ

DLZ's role is to deliver the raw cost data in a queryable form. CUDOS is one of
several possible consumers (Vantage, Kubecost, Looker, custom QuickSight, Grafana,
internal portals). Bundling CUDOS into DLZ would:

- Default-enable QuickSight Enterprise billing (per-user, ongoing)
- Pin customers to QuickSight even if they prefer a different BI tool
- Couple DLZ's release cadence to upstream CUDOS template churn
- Pull SSO / identity-store / user-group concerns into the platform layer

## What DLZ provides for CUDOS

When `finOps.dataExports` is configured with a `STANDARD_CUR_2_0` entry in
`exports`, DLZ delivers everything the CUDOS deployment template needs:

| What CUDOS needs | What DLZ provisions | Where to find it |
|---|---|---|
| CUR 2.0 in Parquet | Shared `dlz-finops-{finOpsAccountId}-{region}` bucket; `finops_standard` Glue table | FinOps account |
| Glue database | `dlz_finops` (configurable via `dataPlaneConfig.glueDatabaseName`) | FinOps account |
| Glue crawler | Daily at 06:00 UTC (configurable); crawls all configured export tables | FinOps account |
| Athena workgroup + results bucket | `dlz-finops` workgroup, results bucket with 30-day expiration ([details](/components/finops/data-exports#athena-workgroup)) | FinOps account |
| Required CUR columns | `includeResources`, `includeSplitCostAllocationData` default to `true` | [`finOps.dataExports.exports.<id>.config`](/components/finops/data-exports#standard_cur_2_0) |

The bucket and Glue database are **shared across all configured export
types** — FOCUS 1.2, Cost Optimization Recommendations, and Carbon Emissions
appear as additional tables (`finops_focus_1_2`, `finops_cost_opt_recs`,
`finops_carbon`) in the same database. CUDOS itself only consumes the CUR
2.0 table; other tools (custom Athena views, BI tools) can join across them.

## Deploying CUDOS against DLZ

CUDOS is deployed in the **FinOps account**, where the data and Glue catalog
live. Two paths:

### Pulling DLZ values from SSM

Both deployment paths take the same DLZ-owned inputs (Glue database, data
bucket, Athena workgroup). DLZ writes them to SSM Parameter Store under
`/dlz/finops/` in the FinOps account — see
[Exported SSM parameters](/components/finops/data-exports#exported-ssm-parameters) for
the full list.

A small shell snippet to load them once, used by both paths below:

```bash
DB=$(aws ssm get-parameter --name /dlz/finops/glue-database-name --query Parameter.Value --output text)
WG=$(aws ssm get-parameter --name /dlz/finops/athena-workgroup-name --query Parameter.Value --output text)
RESULTS=$(aws ssm get-parameter --name /dlz/finops/athena-results-bucket-name --query Parameter.Value --output text)
```

### Path 1 — `cid-cmd` CLI (recommended by AWS)

```bash
pip install --upgrade cid-cmd

# Run from the FinOps account (assume role first)
cid-cmd deploy \
  --athena-database "$DB" \
  --athena-workgroup "$WG" \
  --cur-version 2 \
  --quicksight-user <YOUR-QUICKSIGHT-AUTHOR-USERNAME>
```

The CLI auto-discovers the CUR table inside the database, handles QuickSight
datasource creation, dataset wiring, dashboard deployment, and view
installation. It picks up new dashboards as AWS ships them — no DLZ release
needed.

### Path 2 — AWS Solutions Library CFN template

The [`cid-cfn.yml`](https://github.com/aws-solutions-library-samples/cloud-intelligence-dashboards-framework/blob/main/cfn-templates/cid-cfn.yml)
template can be deployed as a one-shot CloudFormation stack in the FinOps
account. Pass the DLZ-provided values as parameters:

```bash
aws cloudformation deploy \
  --template-file cid-cfn.yml \
  --stack-name dlz-cudos \
  --capabilities CAPABILITY_NAMED_IAM \
  --parameter-overrides \
    DatabaseName="$DB" \
    AthenaWorkgroup="$WG" \
    AthenaQueryResultsBucket="$RESULTS" \
    QuickSightUser=<YOUR-QUICKSIGHT-AUTHOR-USERNAME>
```

The full template parameter list is in the upstream repo; the table below
shows the ones DLZ owns:

| Parameter | DLZ source (SSM) |
|---|---|
| `DatabaseName` | `/dlz/finops/glue-database-name` |
| `AthenaWorkgroup` | `/dlz/finops/athena-workgroup-name` |
| `AthenaQueryResultsBucket` | `/dlz/finops/athena-results-bucket-name` |
| `CurTableName` | `/dlz/finops/exports/standard/glue-table-name` (or `aws glue get-tables --database-name $DB`) |
| `QuickSightUser` | you (DLZ does not manage QuickSight users) |

Updates require redeploying the template. For most teams the CLI is easier.

## Prerequisites you own

DLZ does **not** provision any of these. You provision them once, in the FinOps
account, before deploying CUDOS:

1. **QuickSight Enterprise subscription** (account-level, paid). One-time setup.
2. **At least one QuickSight Author**. CUDOS dashboards are owned and refreshed
   by an Author identity.
3. **(Optional) QuickSight Readers** for everyone who needs to view dashboards
   without editing.
4. **SPICE capacity** sized for your CUR volume.
5. **IAM Identity Center / SSO integration** if you don't want to manage
   QuickSight users manually.

Provision QuickSight in the same region as the CUR bucket where practical —
cross-region SPICE refresh costs more.

## Cost breakdown

<Aside type="note">
This page does not list dollar figures — AWS pricing changes and we don't carry
the maintenance burden of stale numbers. Each section below links to the
authoritative AWS pricing page. Use the [AWS Pricing Calculator](https://calculator.aws/)
to model your specific usage before committing.
</Aside>

### QuickSight — the dominant cost

[AWS QuickSight pricing →](https://aws.amazon.com/quicksight/pricing/)

The cost drivers, in order of impact:

- **Enterprise Authors** (per-user, monthly). Required to build, own, and
  refresh CUDOS dashboards. Annual commit pricing is meaningfully cheaper than
  month-to-month — pick the commit if you're staying. CUDOS typically needs at
  least one Author (the dashboard owner / refresher).
- **Enterprise Readers**. Per-session pricing with a monthly per-reader cap.
  Cheap for occasional viewers; the cap protects heavy users.
- **SPICE capacity**. Each Author includes a small allowance; beyond that it's
  per-GB-month. Sized by how much CUR data you load into SPICE for interactive
  speed.
- **Generative BI (Q)**. Optional add-on. Skip it unless the natural-language
  / narrative-summary features are an explicit ask — it's a significant uplift
  per Author and per Reader.

### Athena — query costs from dashboard refreshes

[AWS Athena pricing →](https://aws.amazon.com/athena/pricing/)

CUDOS runs many SQL queries per refresh (one per view). Cost is driven by:

- **Bytes scanned per query.** Athena bills by data scanned. CUR Parquet with
  columnar pruning makes each refresh scan a small fraction of the dataset.
- **Refresh cadence.** Daily is the CUDOS default; weekly or on-demand cuts
  scans roughly 5–7×.
- **Number of dashboards deployed.** CID includes many; deploying only the
  ones you'll actually use trims both Athena scans and SPICE footprint.

### Other AWS line items — small

- [S3 pricing](https://aws.amazon.com/s3/pricing/) — GETs from Athena scans
  against the shared data bucket. Usually negligible at CUR-data scale.
- [Lambda pricing](https://aws.amazon.com/lambda/pricing/) — the CUDOS view
  manager runs briefly per refresh. Typically a few cents.
- [Glue pricing](https://aws.amazon.com/glue/pricing/) — DLZ already runs the
  shared crawler; CUDOS adds nothing.
- [CloudWatch pricing](https://aws.amazon.com/cloudwatch/pricing/) — log
  ingestion / retention for the workgroup metrics if enabled.

### Modeling your bill

Use the [AWS Pricing Calculator](https://calculator.aws/) with these inputs:

- Number of QuickSight Authors (start with 1)
- Number of Readers + expected sessions per reader per month
- SPICE GB to load (proportional to your CUR volume)
- CUR monthly volume (for Athena scan estimates) and refresh cadence

QuickSight will be the dominant line item for most teams. Athena tends to be
cheap relative to QuickSight unless you refresh aggressively against very large
CUR data.

### Cost-control levers

- **Annual Author commit** is cheaper per month than month-to-month. Pick it if
  the team is staying.
- **Reader session pricing** suits occasional viewers; the per-reader monthly
  cap protects heavy users — most orgs don't need to pre-commit Reader seats.
- **Refresh cadence** — daily is the default; weekly or on-demand cuts Athena
  scans substantially.
- **SPICE only what you query interactively** — direct-query mode skips SPICE
  storage but costs more per query. Trade-off, not always a win.
- **Skip dashboards you won't use.** CID has many; deploy only the ones your
  audience consumes.

## When CUDOS is the wrong choice

- You're using **Vantage / Kubecost / CloudHealth** — they consume the same CUR
  S3 data; don't pay for QuickSight on top.
- You're standardizing on **Looker / Tableau / Grafana / Metabase** — point them
  at the DLZ Glue catalog directly.
- You only need **per-team showback emails** — `finOps.budgets` and
  `finOps.accountBudgets` already do that, no dashboard needed.
- Your CUR volume is small and **AWS Cost Explorer suffices** — Cost Explorer is
  free; deploy CUDOS only when CE's pivot/grouping limits start hurting.

## Update path

CUDOS dashboards evolve continuously. Re-run `cid-cmd update` (or redeploy the
CFN template with a newer version) to pick up new dashboards, fixed views, or
new CUR columns. DLZ does not gate this — your update cadence is independent of
DLZ releases, by design.

## Boundary recap

DLZ stops at the data. CUDOS — and any other BI/visualization layer — is your
choice and your operational responsibility. DLZ keeps the inputs CUDOS needs
stable (bucket name, Glue DB name, CUR column set, Athena workgroup) so deploying
CUDOS against DLZ is a parameter-passing exercise, not an integration project.

The DLZ-owned inputs CUDOS depends on are stable across renames thanks to
construct-id pinning on the Glue database (any future rename forces a clean
CFN Replacement instead of an in-place Update that AWS Glue's API doesn't
support).
